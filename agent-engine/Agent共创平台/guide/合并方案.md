# 合并方案

## 1. 文档目标

本文档用于确定 `muiltchat`、`Cocreator`、`todify4` 三个项目的最终合并落地方案，明确以下内容：

- 最终主仓与总体架构
- 三项目能力映射关系
- 四大功能模块的最终落位
- 前后端目录设计
- 数据库调整建议
- API 与运行时边界
- 分阶段实施路径

本方案的目标不是让三套系统长期并存，而是形成一个统一产品、统一前端、统一后端、统一数据库、统一模型网关的 AI 协作平台。

## 2. 总体结论

- 建议以 `Cocreator` 作为唯一主仓和平台底座。
- `muiltchat` 主要贡献 `AI 对话` 与 `多 AI 对比` 的成熟交互和并发多模型体验。
- `todify4` 主要贡献 `手动 Agent 调用` 与 `多 Agent 讨论` 的运行时编排思路与产品形态。
- 后端统一为 `Node.js + Express + Prisma + SQLite/PostgreSQL` 路线，不保留 `FastAPI` 作为长期主服务。
- 模型调用统一收敛到 `LLM Gateway`，统一管理 provider、模型选择、流式输出、鉴权和失败重试。

一句话定义最终方案：

> 以 `Cocreator` 为平台底座，吸收 `muiltchat` 的聊天与多模型对比能力，吸收 `todify4` 的 Agent Runtime 与 Discussion Runtime，形成一个统一的四模块 AI 协作平台。

## 3. 三项目能力映射

### 3.1 Cocreator 的定位

`Cocreator` 作为最终主仓，负责承载：

- 项目体系
- 用户体系
- 数据库存储
- 知识体系与项目上下文
- Skill/Chain/Execution 资产
- 协作、版本、历史记录
- 统一模型网关
- 统一会话与执行追踪

### 3.2 muiltchat 的定位

`muiltchat` 作为能力来源项目，主要迁入：

- 单模型对话体验
- 多模型并发对比体验
- 聊天消息流与会话交互模型
- 多模型选择与对比展示方式

它不再作为独立系统长期存在，其 Python 后端 provider 实现应迁入统一 Node 网关。

### 3.3 todify4 的定位

`todify4` 作为能力来源项目，主要迁入：

- `AI Role` 到 `AgentDefinition` 的抽象方式
- `AgentOrchestrator` 的手动 Agent 执行 Runtime
- `Brainstorm` 的多 Agent 讨论会话模型
- `parallel / round-robin / debate` 讨论模式
- 主持人、裁判、共识检测、反思循环、结构化上下文
- 工具调用事件流与执行可观测性

它不再作为独立业务系统整体迁入，而是把可复用的运行时思想和产品交互抽离为平台能力。

## 4. 最终产品模块

最终平台保留并产品化以下四大一级模块：

- `AI 对话`
- `多 AI 对比`
- `手动 Agent 调用`
- `多 Agent 讨论`

同时复用 `Cocreator` 现有的知识、项目、协作与执行沉淀能力。

### 4.1 AI 对话

- 单模型聊天
- 流式输出
- 会话持久化
- 上下文连续
- 历史记录检索
- 与项目上下文、来源信息、知识图谱联动

该模块的交互形态主要参考 `muiltchat`。

### 4.2 多 AI 对比

- 同一输入并发调用多个模型
- 同屏展示多个模型输出
- 支持优选、采纳、保存最佳答案
- 支持比较结果回写为项目资产

该模块的交互形态主要参考 `muiltchat`，后端运行时在 `Cocreator` 中重建。

### 4.3 手动 Agent 调用

- 用户手动选择 Agent
- 手动指定模型、提示词、知识来源、上下文窗口
- 可查看工具调用过程、执行日志、输出版本
- 与 Skill、Workflow、知识库联动

该模块产品形态主要参考 `todify4` 的技术包装页，但底层概念以 `AgentDefinition` 为中心，不保留 `featureType + 页面业务映射` 作为核心架构。

### 4.4 多 Agent 讨论

- 支持多个 Agent 围绕同一话题讨论
- 支持 `parallel`、`round-robin`、`debate` 三种模式
- 支持主持人、裁判、共识检测、反思循环、结构化上下文
- 支持轮次控制、消息时间线、最终总结、讨论导出

该模块主要参考 `todify4` 的 `Brainstorm` 体系进行平台化重构。

### 5.2 分层原则

- 产品层：面向用户的四大模块页面
- API 层：按业务域拆分接口
- Runtime 层：按执行形态拆分运行时
- 平台层：统一模型网关、知识上下文、事件流、执行追踪
- 数据层：统一会话、消息、执行、Agent、讨论等资产模型

### 7.3 各 Runtime 的职责

#### Chat Runtime

- 普通聊天
- 会话与消息保存
- 流式输出
- 上下文裁剪
- 单轮和多轮对话管理

#### Compare Runtime

- 多模型并发调用
- 模型结果聚合
- 超时控制与失败降级
- 结果优选与采纳记录

#### Agent Runtime

- 基于 `todify4/AgentOrchestrator` 重构
- 负责 Prompt 渲染、上下文组装、工具调用循环、执行追踪、消息落库
- 统一承载“手动 Agent 调用”

#### Discussion Runtime

- 基于 `todify4/BrainstormOrchestrator` 重构
- 支持并行、轮流、辩论、主持人、裁判、共识、反思、总结
- 统一承载“多 Agent 讨论”

#### Skill Runtime

- 复用 `Cocreator` 现有 Skill 执行能力
- 面向结构化工具、模板能力、可配置动作

#### Workflow Runtime

- 复用和扩展 `Cocreator` 工作流/链式执行能力
- 面向复杂编排、多步链路、自动化流程

## 9. 数据库最终方案

### 9.1 总体原则

- 采用“在现有模型上扩展 + 少量新增表”的方式
- 不推翻现有 `ConversationSession / ConversationMessage / SkillExecution`
- 所有 AI 行为最终都归档到统一 Session/Message/Execution 体系

### 9.2 扩展 `ConversationSession`

新增字段建议：

- `moduleType`：`CHAT | COMPARE | AGENT | DISCUSSION`
- `sessionMode`：`single | multi-model | manual-agent | multi-agent`
- `titleSource`：标题来源
- `selectedModels`：JSON，保存模型列表
- `discussionConfig`：JSON，保存讨论配置
- `linkedExecutionId`：关联执行记录
- `archivedAt`：归档时间

建议形态：

```prisma
model ConversationSession {
  id                String   @id @default(cuid())
  projectId         String
  userId            String
  skillName         String?
  title             String   @default("新会话")
  moduleType        String   @default("CHAT")
  sessionMode       String   @default("single")
  state             String   @default("ACTIVE")
  variables         String   @default("{}")
  config            String   @default("{}")
  selectedModels    String   @default("[]")
  discussionConfig  String   @default("{}")
  linkedExecutionId String?
  isActive          Boolean  @default(true)
  archivedAt        DateTime?
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt
  messages          ConversationMessage[]
}
```

## 11. 核心设计取舍

### 11.1 明确保留与舍弃

保留：

- `muiltchat` 的聊天体验与多模型并发对比体验
- `todify4` 的 `Agent Runtime` 与 `Discussion Runtime`
- `Cocreator` 的项目、知识、Skill、执行资产与平台底座

舍弃：

- `muiltchat` 的 Python 后端长期主服务角色
- `todify4` 的页面业务化 `featureType + pageType + field mapping` 作为核心平台抽象
- 三套数据库与三套 provider 并行维护

### 11.2 最推荐的设计结论

- 聊天与对比直接复用统一 `ConversationSession`
- Agent 必须单独建 `AgentDefinition`
- 多 Agent 讨论必须单独建 `DiscussionSession`
- 所有模型调用必须统一到一个网关
- Skill、Workflow、Agent 是底层运行形态，不能与页面模块概念混用

## 12. 实施路径

### 12.1 Phase 1：统一聊天与模型层

目标：

- 接入 `muiltchat` 的 `AI 对话`
- 接入 `muiltchat` 的 `多 AI 对比`
- 建立统一 `LLM Gateway`
- 扩展 `ConversationSession` 与 `ConversationMessage`

交付物：

- `AI 对话` 可用
- `多 AI 对比` 可用
- 历史记录统一

### 12.2 Phase 2：落地手动 Agent 调用

目标：

- 新增 `AgentDefinition`
- 迁移 `todify4/AgentOrchestrator` 核心逻辑
- 建立 Agent 执行、工具调用事件流、执行追踪

交付物：

- `手动 Agent 调用` 可用
- 支持日志、工具调用、输出版本

### 12.3 Phase 3：落地多 Agent 讨论 MVP

目标：

- 新增 `DiscussionSession` 与 `DiscussionParticipant`
- 迁移 `todify4/BrainstormOrchestrator` 基础能力
- 先支持 `parallel`、`round-robin`
- 支持总结与导出

交付物：

- `多 Agent 讨论 MVP` 可用

### 12.4 Phase 4：增强讨论与治理能力

目标：

- 支持 `debate`
- 支持主持人、裁判、共识检测、反思循环
- 统一回放与审计
- 增强讨论可视化

交付物：

- 生产级讨论编排能力

## 13. 版本目标

- `V1`：聊天、对比、手动 Agent 调用全部跑通，统一历史记录
- `V2`：多 Agent 讨论 MVP，支持 `round-robin` 和最终总结
- `V3`：辩论模式、投票、主持人、裁判、反思循环、讨论可视化

## 14. 最终结论

本次合并的本质不是把三个项目简单拼接，而是进行一次明确的能力收编与平台统一：

- `Cocreator` 提供平台底座
- `muiltchat` 提供聊天与多模型对比体验
- `todify4` 提供 Agent 与讨论的运行时能力

最终产物应是一个统一的 AI 协作平台，而不是三个系统的组合体。

最终推荐路线如下：

- 主仓：`Cocreator`
- 聊天能力来源：`muiltchat`
- Agent 与讨论能力来源：`todify4`
- 统一模型层：`LLM Gateway`
- 统一数据层：`Session + Message + Execution + Agent + Discussion`

这是兼顾当前可落地性、后续可扩展性与长期维护成本的最优方案。
